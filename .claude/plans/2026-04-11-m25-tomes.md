# Milestone 2.5: Tomes (Multi-Project Support)

**Date:** 2026-04-11

---

## Goal

Each campaign/world is a self-contained `.vaelorium` file. The app opens one Tome at a time. A Tome Picker home screen lets users create, open, and switch between Tomes.

**References:**
- Design spec: `design/specs/milestone-2.5-tomes.md`
- Mockups: `design/vaelorium-tomes.pen` (KJ6aC: Tome Picker, 4e6dC: Create New Tome)

## Approaches Considered

### 1. Hot-Swap DbPool via Tauri Managed State
- **Description:** Replace the `DbPool` in Tauri's managed state when opening/closing Tomes. Use `app.manage()` for the initial state and swap via `RwLock<Option<DbPool>>` wrapper.
- **Pros:** Clean separation. No DB when no Tome open. Existing commands work unchanged after swap.
- **Cons:** Need to wrap DbPool in an Option to handle no-Tome state. All existing commands need to handle the None case.

### 2. Single DbPool with ATTACH/DETACH
- **Description:** Keep one app-level DB and ATTACH Tome databases as needed.
- **Pros:** Always have a connection. Can query app state and Tome simultaneously.
- **Cons:** Complex query prefixing. SQLite ATTACH has limits. Harder to reason about.

### 3. Restart App on Tome Switch
- **Description:** Close and relaunch the app process with the Tome path as an argument.
- **Pros:** Simplest — no state swapping.
- **Cons:** Slow. Disruptive UX. Not viable for frequent switching.

## Chosen Approach

**Approach 1: Hot-Swap DbPool.** Wrap the managed state in `RwLock<Option<DbPool>>`. When no Tome is open, it's `None` and all page/entity commands return an error. When a Tome is opened, create a new pool and set it. App-level state (recent Tomes) uses a separate JSON file, not SQLite.

## Notes

- The `.vaelorium` file is just a renamed `.sqlite` database. All existing migrations run on it.
- App state JSON lives in the platform app data dir, outside any Tome.
- The Tauri file dialog plugin (`@tauri-apps/plugin-dialog`) provides native save/open file dialogs.
- For the browser mock, Tome switching just resets the in-memory mock DB.
- The `tome_metadata` table is a simple key-value store inside each Tome database.
- Cover images can be stored as base64 in the metadata table for simplicity (no external file references).
