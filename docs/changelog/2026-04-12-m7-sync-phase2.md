# M7 Sync — Phase 2: Vertical Slice (Pages End-to-End)

**Date:** 2026-04-12

## Summary
Sync now works end-to-end for the `pages` and `page_content` tables: snapshot+journal engine, encrypted upload/download, per-field conflict detection, and a multi-device test harness exercising the 6 scenarios from the plan plus 2 defensive cases. No user-facing UI yet (Phase 3 territory) — but a dev-only Tauri command is wired so the engine can be driven from the dev console.

## Changes

### Features
- **`sync::engine::sync_tome_once`** — full sync cycle: upload local journal → list/download remote ops → decrypt → conflict-detect per field → apply non-conflicted writes → take snapshot if triggered → prune local journal. Returns a structured `SyncOutcome`.
- **`sync::snapshot`** — `take_snapshot` (VACUUM INTO + clear local-only sync tables + encrypt + upload), `restore_snapshot_to_file` (download + decrypt + write), `should_snapshot` (first-snapshot OR ≥5MB OR ≥5000 ops triggers).
- **`sync::journal`** — op builders (`insert_op`, `update_op`, `delete_op`), `record_op` (atomic with caller's tx), `pending_ops` (queue iteration), `prune_journal` (post-snapshot GC), `active_sync_session` (single query that mutation paths use to gate op emission).
- **Per-field conflict detection** with bookkeeping-field denylist (`updated_at`, `created_at`, `created_by`, `updated_by` silently LWW; only user-meaningful fields conflict).
- **`commands/pages.rs` op emission** — all 5 mutation paths (`create_page`, `update_page`, `delete_page`, `save_page_content`, `reorder_pages`) now wrap SQL in a transaction with op recording gated on whether sync is configured. Zero behavior change for un-synced Tomes (one extra `SELECT` per mutation).
- **`commands/sync_dev::sync_dev_filesystem`** — dev-only Tauri command for triggering a filesystem-backed sync from the dev console. Replaced by Settings UI in Phase 3.

### Tests
- **8 new sync integration tests** in `src-tauri/tests/sync_integration.rs`:
  - Scenario A — clean sync propagates inserts
  - Scenario B — sequential edits propagate
  - Scenario C — disjoint-field edits auto-merge (no conflict)
  - Scenario D — overlapping-field edits create one conflict
  - Scenario E — long offline catch-up (20 ops)
  - Scenario F — snapshot bootstrap for fresh device
  - Defense — schema version mismatch parks instead of crashing
  - Defense — wrong passphrase fails to decrypt remote ops
- **5 new sync unit tests** in `sync::snapshot` covering threshold logic and op-builder semantics.
- All 44 vitest + 20 Playwright spot-checks pass; zero regressions.

## Rationale
Phase 2 is the de-risking milestone for M7. With the engine working end-to-end against a real test harness exercising the brainstorm's hard scenarios (long offline, concurrent disjoint, concurrent overlapping, snapshot bootstrap), the architecture is validated. Phases 3–5 are now mostly mechanical: UI surface, broaden to other tables, polish.

## Discoveries
- **Bookkeeping fields demand a denylist.** `updated_at`/`created_at`/etc. get touched on every write. Without filtering them out of conflict detection, every concurrent edit produced false-positive conflicts. Codified as `engine::is_meta_field()`.
- **Snapshot DB must open without running migrations.** Some existing migrations (e.g. `005_relations.sql`'s `CREATE INDEX` statements) lack `IF NOT EXISTS`, so re-applying them on a snapshot-restored DB fails. Snapshot restore should open the file directly and trust its embedded schema. Production restore path needs the same care.
- **Op-emission overhead is below noise.** One extra `SELECT … LIMIT 1` per mutation when sync is disabled. Worth measuring more rigorously in Phase 5 polish.
