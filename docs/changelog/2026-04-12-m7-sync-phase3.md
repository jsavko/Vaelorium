# M7 Sync — Phase 3: Settings UI, Conflict Resolver, Background Runner

**Date:** 2026-04-12

## Summary
Sync is now usable from the UI. New `Settings → Sync` tab with backend setup flow, sidebar status pill that always reflects current state, inline `ConflictResolver.svelte` for per-field resolution, and a Tokio background runner that ticks every 5 minutes and is nudged within ~10s of any mutation. Replaces the dev-only Phase 2 command with 8 public Tauri commands.

## Changes

### Backend
- **`sync::session::SessionState`** — Tauri-managed in-memory cache for the unlocked `KeyMaterial` + a `Notify` handle for mutation nudges. Cleared on disable; never persisted.
- **`sync::runner`** — long-lived Tokio task: 5-min poll OR debounced nudge wakes it, runs `sync_tome_once`, emits `sync:status` Tauri events. Cancellable through the session lifecycle.
- **`commands/sync`** — 8 new public Tauri commands replacing the dev-only `sync_dev_filesystem`:
  - `sync_enable` (idempotent, validates backend, derives + caches key)
  - `sync_disable`, `sync_now`, `sync_status`, `sync_take_snapshot`, `sync_unlock`
  - `sync_list_conflicts`, `sync_resolve_conflict` (applies choice + emits resolution op + clears conflict row)
- **`commands/pages.rs`** — every mutation that records ops now also calls `session.nudge()` after commit, so the runner debounces a sync within ~10s.
- **`commands/sync_dev.rs`** — deleted.

### Frontend
- **`src/lib/api/sync.ts`** — typed wrapper for the 8 commands + `subscribeSyncStatus()` event listener.
- **`src/lib/stores/syncStore.ts`** — Svelte stores: `syncStatus`, `syncConflicts`, `syncRunning`, derived `syncIndicator` (idle / syncing / conflicts / offline / error). `initSyncStore()` mounts the event subscription.
- **`Settings.svelte`** — new `Sync` tab between Data and Account:
  - Off state: explanatory text + "Enable sync" button.
  - Setup flow inline: backend picker (filesystem only; S3 disabled with "coming next" hint), folder picker via Tauri dialog, passphrase + confirm with the "Lose this passphrase and your data is unrecoverable. There is no recovery." warning, device name, error surfacing.
  - Enabled state: status card (backend, device, last sync, queue size, conflict count, error), three actions (Sync now / Take snapshot / Disable sync).
- **`Sidebar.svelte`** — sync pill below the header. Color-coded dot + label, click opens Settings. Pulses while syncing.
- **`ConflictResolver.svelte`** — new component, mounted above MainContent in App.svelte. Filters conflicts to the active page. Per-field side-by-side picker with radio choices, "Apply resolution" applies all, then refreshes status.

### Tests
- **Playwright e2e** added: `sync tab shows desktop-only notice in browser` — verifies the tab renders and the desktop-only fallback works in the browser build (the only path Playwright can exercise without Tauri).
- All 38 Rust unit tests, 8 Rust integration tests, 44 vitest, 8 Playwright settings tests pass; no regressions.

## Rationale
Phase 2 made sync work; Phase 3 makes it visible. The runner shape (slow poll + mutation nudge) directly answers the user-stated workflow: "sometimes I work on my laptop, sometimes my pc; sometimes I have internet, sometimes I don't." Background sync handles the "remote changes I haven't asked about" case; the nudge handles the "I just edited and want to switch devices in 30 seconds" case.

The conflict resolver is per-page intentionally — most worldbuilders touch one page at a time, and showing only the conflicts on the visible page keeps the UI uncluttered. Cross-page bulk resolution waits for Phase 5 polish.

## Known Limitations (Deferred)
- **Single-Tome scope.** The runner only syncs the currently-open Tome. Multi-Tome background sync needs a per-Tome pool registry; that's an architectural lift for Phase 5 or later.
- **Passphrase re-entry on app restart.** Cached key is process-local; closing the app forgets it, requiring `sync_unlock` (no UI yet — call from devtools to validate). OS keychain integration is the natural fix; landing later.
- **`sync_resolve_conflict` `before` is approximate.** We pass the rejected value as the "before" of the resolution op rather than tracking the true common-ancestor value. Functionally correct for LWW propagation but theoretically loses some merge information. Revisit if it surfaces.
- **No first-run wizard.** Users find Sync via the Settings tab; a guided onboarding flow lands in Phase 5.
