# M7 Sync — Phase 1: Engine Spine

**Date:** 2026-04-12

## Summary
First commit of M7 sync infrastructure: typed op format, E2EE primitives (Argon2id + ChaCha20-Poly1305), backend abstraction trait + filesystem implementation, per-Tome sync state tables. No user-facing changes yet — this is the foundation Phase 2 builds on.

## Changes

### Features
- **`sync::journal::Op`** — typed mutation record (insert/update/delete + table + row + before/after fields + ULID op_id + transaction_id) for the append-only delta journal.
- **`sync::crypto`** — passphrase → 32-byte key via Argon2id (OWASP defaults: m=64MiB, t=3, p=1), AEAD via ChaCha20-Poly1305 with random per-message nonces. Wire format: `nonce(12) || ciphertext+tag`. Key bytes zeroed on drop.
- **`sync::backend::SyncBackend`** trait — async object-store interface with `put/get/list/delete/head/atomic_swap`. Etag-based CAS for snapshot-pointer races.
- **`sync::backend::FilesystemBackend`** — local-directory implementation; SHA-256 content-addressable etags; atomic-rename writes; in-flight tempfile filtering on list.
- **`sync::state::SyncConfig` + `SyncRuntimeState`** — sqlx-backed structs for per-Tome configuration (backend type, salt, device identity) and runtime state (last uploaded/applied op IDs, last sync time).
- **`sync::conflict::Conflict`** — descriptor for unresolved per-field conflicts (consumed by Phase 3's ConflictResolver UI).
- **Migration `009_sync.sql`** — adds `sync_config`, `sync_state`, `sync_journal_local`, `sync_conflicts` tables.

### Tests
- 33 new Rust unit tests in `sync::*` covering: op (de)serialization including delete semantics, KDF determinism + key separation by passphrase and salt, encrypt/decrypt roundtrip, wrong-passphrase / tampered-ciphertext / malformed-blob rejection, fresh-nonce-per-encryption, filesystem backend put/get/list/delete/head/CAS, etag determinism, in-flight tempfile filtering, conflict descriptor roundtrip, sync_config + sync_state DB roundtrips and upserts.

## Rationale
Worldbuilders need cross-device sync (M7 in the roadmap). The brainstorm and plan picked a snapshot-baseline + encrypted-delta-journal model with E2EE, pluggable backends (filesystem + S3), and inline conflict resolution. Phase 1 establishes the engine primitives (op format, encryption, backend interface, state tables) without yet wiring anything into existing mutation paths — keeping the change reviewable and the rest of the app untouched.

## Deviations
- **`age` → `chacha20poly1305 + argon2`.** age's passphrase mode uses scrypt internally; combining it with our chosen Argon2id KDF was awkward. The brainstorm explicitly named this fallback (`"swap to chacha20poly1305 + custom framing"`), and our framing is just `nonce(12) || ciphertext+tag` — minimal.
- **`aws-sdk-s3` not pinned now** as the plan suggested — defers compile-time bloat to Phase 4 where the dep actually gets used. Will pin a stable major then.

## Footgun discovered
The production migration runner (`db::migrations::split_sql_statements`) joins multi-line SQL with spaces and strips full-line `--` comments, but **does not strip inline `--` comments mid-line** — which then eat the rest of the statement after joining, producing SQLite "incomplete input" errors. The migration was rewritten to put comments on their own lines. Future migrations should follow the same convention until the splitter is hardened.
